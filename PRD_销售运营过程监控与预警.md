## 1. 文档说明

本 PRD 以 2026-04-18 当前仓库中的最新 H5 页面为准，页面已经从“销售运营过程监控与预警”进一步收敛为一个**管理视角的销售运营看板**。

本次更新的重点不是补更多字段，而是确保 PRD 与当前页面实际结构一致，避免继续沿用旧版文档中的独立“口径说明区”“预警完成统计区”“业务关注字段区”等已下线结构。

## 2. 背景与目标

### 2.1 背景

业务负责人需要的不是一个大而全的 BI 平台，而是一页内完成以下判断：

1. 当前期次整体盘面是否健康
2. 转化断点主要卡在哪个环节
3. 哪些销售该冲、该稳、该救
4. 今天优先该推动哪些管理动作

因此，页面需要把“结果、趋势、结构、责任、动作”放到同一条阅读链路中，而不是把经营数据和执行数据拆成多个割裂模块。

### 2.2 产品目标

1. 让负责人在 3 分钟内看清当前期次整体经营状态
2. 让负责人快速定位链路断点和成交贡献结构
3. 让负责人可以直接完成销售分层和目标调整判断
4. 让页面从“看数”直接过渡到“定动作”

### 2.3 非目标

- 不建设多页面经营分析平台
- 不在本期展开客户级明细下钻
- 不在本期建设预警规则配置后台
- 不在本期补充独立的企微销售工作台流程

---

## 3. 目标用户与核心场景

### 3.1 目标用户

- 销售负责人
- 运营负责人
- 体验营项目负责人

### 3.2 核心场景

1. 早会前查看当前期次整体盘面
2. 周中判断该补承接还是抓成交
3. 复盘时识别渠道贡献和销售贡献差异
4. 依据执行状态与短板判断销售资源倾斜方向

---

## 4. 页面定位

该页面是一个**后台管理型销售运营看板**，服务对象是负责人，而不是一线销售。

页面输出不止是指标展示，还必须支持 4 个管理动作：

1. 看整体结果
2. 找关键断点
3. 定销售分层
4. 给出本期动作建议

---

### 5.1 本期范围

## 6. 数据与口径原则

### 6.1 主口径原则

页面中所有核心经营判断围绕当前选中的体验营期次展开，主展示逻辑以当前页面已有指标为准。

### 6.2 页面当前展示指标池

- 总添加人数
- 有效 leads
- SA 客户数
- 领卡人数
- 填问卷人数
- 直播间人数
- 转化人数
- 退款人数
- 未回复人数
- 付款人数
- 成交人数

### 6.3 口径要求

1. 同一指标在卡片、漏斗、趋势、表格中的名称和含义必须一致
2. 页面中的说明以悬浮提示和模块说明文案为主，不再单独拆出“统计口径说明区”
3. 所有趋势图、占比图、销售表均随顶部筛选条件联动
4. 当前期次是核心分析维度，其他维度只做辅助筛选

---

## 7. 信息架构

当前 H5 的页面结构固定为：

1. 顶部筛选区
2. 第一屏：经营概览 + 转化漏斗
3. 第二屏：转化趋势
4. 第三屏：客户成交来源分布 + 客户成交占比
5. 第四屏：销售作战视图
6. 第五屏：销售数据明细

页面阅读逻辑为：

1. 先看整体盘面
2. 再看转化断点
3. 再看趋势和结构
4. 再看销售分层与目标动作
5. 最后看销售明细落实责任

## 8. 模块设计

### 8.1 顶部筛选区

#### 模块目标

统一控制全页口径，保证页面所有图表和表格在同一分析视角下联动。

#### 当前页面内容

- 筛选提示文案：核心筛选维度为体验营期次
- 期次筛选
- 销售筛选
- 客户来源筛选
- 时间范围快捷筛选
- 开始日期
- 结束日期

#### 筛选要求

1. 期次为最核心筛选项，默认进入当前期次
2. 全页图表、卡片、表格必须联动
3. 时间范围支持快捷项与日期区间组合
4. 筛选数量保持克制，避免影响负责人读取效率

---

### 8.2 经营概览

#### 模块目标

在页面第一屏先回答“这一期整体经营结果怎么样”。

#### 当前展示卡片

第一行：

- 总添加人数
- 有效 leads
- SA 客户数
- 领卡人数

第二行：

- 填问卷人数
- 直播间人数
- 入群人数（**去重口径**：同一客户加多个群只计 1 人）
- 转化人数
- 退款人数
- 未回复人数

#### 每张卡片的展示规则

1. 指标名
2. 指标说明悬浮提示
3. 当前值
4. 一句简短业务解释
5. 环比
6. 同比

#### 模块要求

1. 概览卡必须同时覆盖规模、承接、成交、风险四类信息
2. 风险指标需通过风险色突出，例如退款人数
3. 未回复人数虽为下降趋势，也需要保留风险语义提示
4. 卡片说明文案必须帮助负责人快速理解该指标管理意义，而不是重复字段名

---

### 8.3 转化漏斗

#### 模块目标

回答“转化链路主要卡在哪”。

#### 当前漏斗层级

1. 总添加
2. 有效 leads
3. 领卡
4. 问卷
5. 付款人数
6. 成交

#### 每层展示内容

- 当前人数
- 相对上一层转化率
- 当前层流失量

#### 模块要求

1. 漏斗必须突出最大掉点环节
2. 文案默认服务于负责人，不强调技术性图表表达
3. 若图表库不可用，需要有 fallback 文案
4. 漏斗层级必须与经营概览、销售表中的关键字段保持一致

---

### 8.4 转化趋势

#### 模块目标

回答“当前期次表现放在近期周期里是什么水平”，为目标校准提供参照。

#### 当前趋势维度

- X 轴：体验营期次
- Y 轴：人数

#### 当前展示序列

- 问卷数
- 转化数
- 退款数

#### 模块要求

1. 该区用于判断节奏，不替代概览区
2. 趋势图应支持负责人快速识别上升、停滞、风险上升三种信号
3. 问卷数与转化数属于核心结果序列，退款数属于风险序列

---

### 8.5 成交结构分析

本区由两个并列图表组成，用于回答“成交来自哪里”和“谁贡献了成交”。

#### A. 成交客户添加渠道

**图表类型**：双层嵌套环形图（旭日图）

一级渠道（内圈）：
- 订阅号/公众号 · 视频号 · 网站 · 矩阵服务号 · 合作引流 · 深海圈 · 小程序 · 长尾

二级子渠道（外圈，按需扩展）：
- 网站 → 索引站 / 搜索引擎
- 合作引流 → 用户分销 / KOC导流
- 深海圈 → 深海圈AI编程 / 深海圈B站好物

交互规则：
- 标签全部隐藏，hover 显示渠道名 + 总计 + 二级明细占比卡片
- 圆心动态显示 hover 渠道名及成交量
- 右侧图例展示一级渠道名 + 成交数量
- 底部注明「内圈为一级渠道，外圈为二级渠道」

模块目标：判断添加渠道结构是否健康，是否过度依赖单一渠道

#### B. 客户成交占比

**图表类型**：双轴柱状图

- 左轴柱状：按销售人员统计成交单量
- 右轴折线：各销售转化率（%）

模块目标：看销售成交贡献与转化效率，判断是否存在头部过度集中

#### 模块要求

1. 两个图表默认放在同一阅读层级
2. 渠道图看来源结构，销售占比图看责任归属，两者不能混用
3. 图例和 tooltip 必须让负责人直接读懂”成交单数 + 占比/转化率”

---

### 8.6 销售作战视图

#### 模块目标

这是页面的管理输出核心区，不只是展示数据，而是明确告诉负责人：

1. 团队当前总诊断是什么
2. 本期该冲谁
3. 本期该救谁
4. 本期该盯什么
5. 目标还差多少
6. 销售应如何分层

#### 当前模块结构

第一层：4 张诊断卡

- 销售团队总诊断
- 本期该冲谁
- 本期该救谁
- 本期该盯什么

第二层左侧：3 张目标卡

- 问卷目标
- 付款目标
- 执行目标

第二层右侧：3 张销售分层卡（个人维度）

- 冲刺型销售
- 观察型销售
- 风险型销售

第二层底部：1 条团队执行风险摘要行（团队维度）

- 汇总展示延期待办数 + 未完成预警数
- 以警告色独立呈现，不与个人分层卡混排

> **设计原则**：个人分层卡（冲刺 / 观察 / 风险）是销售维度的分类，团队执行风险是团队维度的汇总，两者语义不同，不应放在同一网格内并列。

#### 目标值来源

目标卡中的目标值（问卷目标、付款目标、执行目标）由**系统基于历史数据自动测算**，取近 3 期均值作为本期基准目标。

- 目标卡默认标注「系统估算」角标，让负责人清楚数值来源
- **目标值支持手动修改**：点击目标数字进入内联编辑状态，Enter / 失焦保存，Esc 取消
- 手动修改后角标变为「手动修改」（绿色），差额文案和追赶建议自动根据新目标值重算
- 超额完成时文案自动切换为「已超额完成 X」

#### 当前页面输出方式

每张卡不只显示数值，还必须同时显示一段明确动作解释，例如：

- 还差多少（含差额数字）
- 优先补什么
- 资源向谁倾斜
- 哪些销售适合承接高潜客户
- 哪些问题应该先修节奏再谈扩量

#### 模块要求

1. 所有结论必须是可执行的管理判断，不能只报数
2. “该冲谁 / 该救谁 / 该盯什么”必须明确到人或明确到关键变量
3. 目标卡必须同时展示目标值、当前值、差额和追赶建议
4. 销售分层卡（个人维度）必须与销售明细表的经营分组保持一致
5. 团队执行风险摘要必须分别列出延期待办数和未完成预警数，不能只展示合计

---

### 8.7 销售数据明细

#### 模块目标

将销售维度的经营结果、执行状态、承接效率和动作建议放进同一张责任表中，支持负责人快速做分配和跟进。

#### 当前表头结构

固定列：

- 销售
- 经营分组
- 当前短板

关键结果：

- 有效 leads
- SA客户
- 领卡数
- 问卷数
- 直播间人数
- 进群人数
- 转化数
- 退款数

执行状态：

- 预警处理
- 平均耗时
- 完成率

承接转化：

- 领卡率
- 问卷率
- 付款率

收尾列：

- 本期建议动作
- 目标调整建议

#### 跳转规则

以下列的数值支持点击跳转至客户管理页面，并传入对应销售人员 + 客户状态过滤参数：

- **有效 leads · SA客户 · 领卡数 · 问卷数 · 转化数**

以下列暂不支持跳转：

- 直播间人数 · 进群人数 · 退款数

#### 当前行级表达方式

每个销售都需要同时体现：

1. 姓名和一句简短状态描述
2. 经营分组标签
3. 当前最主要短板
4. 关键结果数据
5. 执行状态数据
6. 承接转化效率
7. 本期建议动作
8. 目标调整建议

#### 模块要求

1. 表格不是纯数据表，而是责任归属表
2. 经营分组必须使用明显标签样式区分冲刺型、观察型、风险型
3. “当前短板”“本期建议动作”“目标调整建议”必须能支撑负责人直接安排动作
4. 表格字段顺序遵循当前 H5，不再沿用旧版 PRD 中“转化结果 -> 待办完成 -> 用户承接”的旧顺序

## 9. 交互与展示要求

### 9.1 交互要求

1. 筛选后全页联动
2. 图表库不可用时需展示 fallback 文案
3. 核心指标说明通过 hover 提示承载
4. 页面默认按照“概览 -> 断点 -> 趋势 -> 结构 -> 动作 -> 明细”的顺序浏览

### 9.2 视觉要求

1. 风险、正常、向好状态需要清晰区分
2. 第一屏必须让负责人快速进入状态
3. 卡片与表格信息密度高，但要保持可读性
4. 销售分层、目标卡、诊断卡应形成清晰视觉层级

## 10. 与旧版 PRD 的差异说明

以下内容在旧版 PRD 中是独立模块，但在最新 H5 中已调整，不应继续作为独立验收项存在：

- 已取消独立“业务侧关注字段”模块
- 已取消独立“统计口径说明”模块
- 已取消独立“销售预警完成统计”模块
- 已取消独立“今日优先动作区”

对应变化为：

- 业务关注内容已吸收到经营概览、趋势图和销售明细中
- 口径说明改为卡片 tooltip 与模块文案承载
- 预警处理以销售表中的“预警处理”字段和作战视图中的“团队执行风险”表达
- 动作建议吸收到诊断卡、目标卡和销售表建议列中

- 顶部筛选区存在且全页联动

- 转化趋势模块存在
- 成交来源分布与销售成交占比并列展示
- 销售作战视图存在

- 可直接基于页面内容安排本期动作和目标调整

## 13. 结论

最新 H5 页面已经明确收敛为一个管理者使用的销售运营看板，其核心不是展示更多数据，而是把经营判断、销售分层和动作建议放到同一页内完成。

本 PRD 已按 2026-04-18 的 H5 实际结构更新，后续若页面继续迭代，应以当前模块边界为基础增量调整，而不是回退到旧版分散式信息架构。
