# PRD：数据权限与功能权限体系

> **产品**：生财有术 SCRM
> **版本**：v1.0
> **状态**：草稿
> **作者**：明策（PM）
> **创建日期**：2026-03-19
> **最后更新**：2026-03-19

---

## 1. 背景与目标

### 1.1 背景

生财有术 SCRM 当前的权限系统仅实现了**基础功能权限**——以"菜单 × 部门"为粒度，控制哪些部门可以访问哪些页面。这种模型存在以下问题：

| 问题 | 说明 |
|------|------|
| **无数据隔离** | 销售 A 可以看到销售 B 的客户数据，交付人员也能看到所有客户 |
| **无字段级控制** | 销售字段（客户来源、跟进状态）和交付字段（会员状态、到期时间）对所有人可见 |
| **无操作级控制** | 无法区分"可查看"和"可编辑"——有菜单权限就能做所有操作 |
| **粒度太粗** | 以部门为单位，无法支持同部门内不同岗位的差异化权限 |
| **无主管视角** | 团队主管无法查看下属的客户数据 |

### 1.2 目标

构建 **RBAC（基于角色的访问控制）+ 数据范围** 的权限体系：

1. **数据行级权限**：不同角色只能看到自己职责范围内的客户数据
2. **字段级权限**：销售侧和交付侧的敏感字段相互隔离
3. **功能权限**：精细到"模块 × 操作"粒度（查看/编辑/删除/导出）
4. **数据范围**：支持个人/团队/部门/全公司四级数据可见范围
5. **向下兼容**：现有的"菜单 × 部门"配置作为功能权限的基础，平滑升级

### 1.3 不做什么

- 不做审批流（权限申请/审批）——管理员直接配置
- 不做临时权限（限时授权）——当前阶段不需要
- 不做 API 级鉴权设计——本 PRD 聚焦产品层面，技术实现另出方案

## 2. 现状分析

### 2.1 当前权限模型

```
┌─────────────┐
│  菜单页面     │ ← 权限对象（约 40+ 页面）
├─────────────┤
│  部门        │ ← 权限主体（6 个部门）
├─────────────┤
│  可用/不可用  │ ← 控制粒度（二值）
└─────────────┘
```

**现有 6 个部门**（来自截图）：
- 航海家运营组
- 会员运营部
- 实战运营部
- 联合办公孵化空间
- 综合管理部^G财务组
- 社区内容部

### 2.2 问题场景

| 场景 | 现状 | 期望 |
|------|------|------|
| 销售 A 查看客户列表 | 能看到所有客户，包括别人的 | 只看到自己的客户（自己是专属销售的） |
| 交付人员看客户详情 | 能看到销售字段（客户来源等） | 只看到交付相关字段（会员状态、到期时间） |
| 销售主管看团队数据 | 和普通销售一样，无团队视角 | 能看到下属所有客户数据（只读） |
| 运营总监看全局数据 | 需要管理员账号 | 有独立角色，可看全公司数据 |
| 实习生做数据录入 | 和正式员工权限一样 | 只能编辑，不能导出和删除 |

## 3. 权限模型设计

### 3.1 整体架构

```
┌──────────────────────────────────────────────────┐
│                    权限引擎                        │
├──────────┬───────────┬───────────┬────────────────┤
│ 功能权限  │ 数据行权限  │ 字段权限   │  操作权限       │
│ 能看哪些  │ 能看哪些   │ 能看哪些   │  能做哪些       │
│ 菜单/页面 │ 数据行     │ 字段列     │  操作           │
├──────────┴───────────┴───────────┴────────────────┤
│                  角色（Role）                       │
│          一个用户可以拥有多个角色                      │
├──────────────────────────────────────────────────┤
│               用户（SCRM 成员）                     │
└──────────────────────────────────────────────────┘
```

### 3.2 权限四层模型

| 层级 | 控制维度 | 回答的问题 | 示例 |
|------|---------|-----------|------|
| **L1 功能权限** | 菜单/页面 | 能不能进这个页面？ | 销售不能进"标签管理"页面 |
| **L2 数据行权限** | 数据范围 | 进了页面能看到哪些数据？ | 销售只能看自己的客户 |
| **L3 字段权限** | 字段/列 | 看到的数据里哪些字段可见？ | 交付看不到"客户来源"字段 |
| **L4 操作权限** | 增删改查导出 | 对可见数据能做什么操作？ | 实习生不能导出和删除 |

**判定流程**：

```
用户访问资源
  │
  ├─ L1: 有功能权限？ ─── 否 → 页面不可见 / 403
  │         │
  │        是
  │         │
  ├─ L2: 数据范围过滤 ── 只返回权限范围内的数据行
  │         │
  ├─ L3: 字段过滤 ────── 隐藏无权限的字段列
  │         │
  └─ L4: 操作校验 ────── 禁用无权限的操作按钮
```

## 4. 角色定义

### 4.1 系统预置角色

> 系统预置角色不可删除，但可以修改其权限配置。另外支持自定义角色。

| 角色代码 | 角色名称 | 业务定位 | 典型人员 |
|---------|---------|---------|---------|
| `super_admin` | 超级管理员 | 系统最高权限，不受权限控制 | 技术负责人 |
| `admin` | 管理员 | 权限配置、系统设置 | 运营总监 |
| `sales` | 销售 | 客户获取与转化 | 销售顾问 |
| `sales_leader` | 销售主管 | 销售团队管理 | 销售组长 |
| `delivery` | 交付 | 会员运营与服务 | 社群运营 |
| `delivery_leader` | 交付主管 | 交付团队管理 | 运营组长 |
| `operator` | 运营 | 内容运营、活动运营 | 内容编辑、活动策划 |
| `analyst` | 数据分析 | 数据看板、报表导出 | 数据分析师 |
| `viewer` | 只读观察者 | 只看不操作 | 实习生、外部顾问 |

### 4.2 角色与部门的关系

```
部门（Department）──── 组织架构，决定"人在哪里"
       │
       │  N:N
       │
角色（Role）────────── 权限集合，决定"人能做什么"
       │
       │  N:N
       │
用户（User）────────── SCRM 系统成员
```

- **一个用户可以有多个角色**：比如小张既是"销售"又是"运营"
- **一个部门的成员可以有不同角色**：会员运营部里可能有"交付"和"交付主管"
- **角色决定权限，部门决定组织归属**：权限跟着角色走，不跟部门走

### 4.3 角色继承

```
super_admin（不受限制）
  └── admin
        ├── sales_leader ──→ 继承 sales 的所有权限 + 团队数据范围
        │     └── sales
        ├── delivery_leader ──→ 继承 delivery 的所有权限 + 团队数据范围
        │     └── delivery
        ├── operator
        ├── analyst
        └── viewer
```

> **继承规则**：子角色的权限 ⊆ 父角色的权限。给子角色新增权限不影响父角色。
> **多角色合并**：用户拥有多个角色时，取各角色权限的**并集**。

## 6. 数据行权限（L2）

### 6.1 数据范围级别

| 级别 | 代码 | 含义 | 数据范围 |
|------|------|------|---------|
| **个人** | `self` | 只看自己的数据 | 当前用户是专属销售/交付人员的客户 |
| **团队** | `team` | 看自己 + 直属下级的数据 | 当前用户及其直属下属名下的客户 |
| **部门** | `dept` | 看本部门所有人的数据 | 当前用户所在部门全体成员名下的客户 |
| **全公司** | `all` | 看所有数据 | 无过滤 |

### 6.2 数据归属规则

客户数据的归属基于**企微好友关系**：

| 好友关系类型 | 归属人 | 角色可见性 |
|------------|--------|-----------|
| 销售好友 | 专属销售 | `sales` / `sales_leader` 可见 |
| 官网交付好友 | 官网交付人员 | `delivery` / `delivery_leader` 可见 |
| 航海家交付好友 | 航海家交付人员 | `delivery` / `delivery_leader` 可见 |
| 深海圈交付好友 | 深海圈交付人员 | `delivery` / `delivery_leader` 可见 |

**关键规则**：
- 一个客户可以同时有销售好友和交付好友关系
- 销售角色看到的客户列表 = 以销售好友关系为维度的数据
- 交付角色看到的客户列表 = 以交付好友关系为维度的数据
- 同一客户在两个角色的视角下，展示不同字段和关注点

### 6.3 预置角色 × 数据范围

| 角色 | 客户管理 | 跟进记录 | 人群包 | 数据看板 |
|------|---------|---------|--------|---------|
| `super_admin` | 全公司 | 全公司 | 全公司 | 全公司 |
| `admin` | 全公司 | 全公司 | 全公司 | 全公司 |
| `sales` | 个人（销售维度） | 个人 | — | — |
| `sales_leader` | 团队（销售维度） | 团队 | 团队 | 团队 |
| `delivery` | 个人（交付维度） | 个人 | — | — |
| `delivery_leader` | 团队（交付维度） | 团队 | 团队 | 团队 |
| `operator` | 全公司（只读） | — | 全公司 | 全公司 |
| `analyst` | 全公司（只读） | — | 全公司 | 全公司 |
| `viewer` | 部门（只读） | — | 部门 | 部门 |

### 6.4 数据范围计算逻辑

```
# 伪代码：计算用户可见客户列表
function getVisibleCustomers(user, role):
  scope = role.dataScope["customer"]  # self / team / dept / all

  if scope == "self":
    if role.type == "sales":
      return customers WHERE sales_owner == user.id
    elif role.type == "delivery":
      return customers WHERE delivery_owner == user.id

  elif scope == "team":
    teamMembers = getDirectReports(user.id)  # 直属下级
    teamMembers.add(user.id)
    if role.type == "sales":
      return customers WHERE sales_owner IN teamMembers
    elif role.type == "delivery":
      return customers WHERE delivery_owner IN teamMembers

  elif scope == "dept":
    deptMembers = getDeptMembers(user.deptId)
    return customers WHERE sales_owner IN deptMembers
                        OR delivery_owner IN deptMembers

  elif scope == "all":
    return customers  # 无过滤
```

### 6.5 跨角色数据可见性

当一个用户拥有多个角色时（如既是销售又是运营），取并集：

```
用户小张的角色：sales + operator
  → sales 数据范围：个人（销售维度）→ 返回 50 个客户
  → operator 数据范围：全公司（只读）→ 返回 2000 个客户
  → 合并结果：2000 个客户（operator 范围更大）
  → 但操作权限：50 个客户可编辑（sales），其余只读（operator）
```

## 7. 字段权限（L3）

### 7.1 字段分组

客户数据的字段按业务归属分为三组：

| 字段组 | 包含字段 | 默认可见角色 |
|--------|---------|------------|
| **共享字段** | 昵称、头像、手机号、星球编号、客户等级、客户状态、企微标签、生命周期阶段 | 所有角色 |
| **销售字段** | 客户来源、专属销售、销售好友状态、跟进状态、最近跟进时间、体验营信息、转化漏斗阶段 | `sales` / `sales_leader` / `admin` |
| **交付字段** | 会员状态、会员类型、会员到期时间、专属交付人员、活跃度指标、新人进度、AI 画像、沉默天数 | `delivery` / `delivery_leader` / `admin` |

### 7.2 敏感字段

以下字段需要额外的权限控制：

| 字段 | 敏感级别 | 默认可见 | 说明 |
|------|---------|---------|------|
| 手机号 | 高 | `admin` + 明确授权 | 脱敏展示（138****1234），完整查看需授权 |
| 聊天记录 | 高 | `admin` + 明确授权 | 需开启会话存档且有查看权限 |
| 订单金额 | 中 | `sales_leader` + `admin` | 普通销售默认不可见 |
| 客户来源渠道 | 中 | `sales` + `admin` | 交付角色默认不可见 |

### 7.3 字段权限在各模块的表现

**客户列表页**：

| 场景 | 可见列 | 不可见列 |
|------|-------|---------|
| 销售查看客户列表 | 共享字段 + 销售字段 | 交付字段隐藏 |
| 交付查看客户列表 | 共享字段 + 交付字段 | 销售字段隐藏 |
| 管理员查看客户列表 | 全部字段 | 无 |

| Tab | 销售可见 | 交付可见 | 管理员可见 |
|-----|---------|---------|-----------|
| 客户档案 | 基础信息 + 销售企微关系 | 基础信息 + 交付企微关系 | 全部 |

## 8. 操作权限（L4）

### 8.1 标准操作类型

| 操作代码 | 操作名称 | 说明 |
|---------|---------|------|
| `view` | 查看 | 读取数据 |
| `create` | 创建 | 新建记录 |
| `edit` | 编辑 | 修改已有记录 |
| `delete` | 删除 | 删除记录（软删除） |
| `export` | 导出 | 导出 CSV/Excel |
| `assign` | 分配 | 变更归属人 |
| `import` | 导入 | 批量导入数据 |

### 8.2 操作权限与数据范围的交叉

操作权限受数据范围约束：

| 数据范围 | 可查看 | 可编辑 | 可删除 |
|---------|--------|--------|--------|
| 个人范围内 | ● | ● | 需 delete 权限 |
| 团队范围内（他人的） | ● | 需 edit 权限 | 需 delete 权限 |
| 全公司范围内（他人的） | ● | 需 edit 权限 | 需 delete 权限 |

**主管编辑规则**：
- 销售主管可以查看下属客户（只读），但**编辑**需要额外的 `team_edit` 权限
- 默认主管只有**查看**权限，编辑权限需显式授予
- 这样设计是为了避免主管误改下属正在跟进的客户数据

## 10. 业务场景走查

### 10.1 场景：销售张三查看客户列表

1. 张三角色：`sales`
2. **L1 功能权限**：客户列表-查看 ✓、编辑 ✓、导出 ✗
3. **L2 数据范围**：个人（销售维度）→ 只返回专属销售 = 张三的客户（共 48 人）
4. **L3 字段权限**：共享字段 ✓、销售字段 ✓、交付字段 ✗ → 列表不显示"会员到期时间"等交付列
5. **L4 操作权限**：查看 ✓、编辑 ✓、导出 ✗ → 导出按钮灰色禁用
6. **结果**：张三看到 48 行客户数据，有销售相关字段，可编辑但不可导出

### 10.2 场景：销售主管李四查看团队数据

1. 李四角色：`sales_leader`
2. **L1**：客户列表全部权限 ✓
3. **L2**：团队（销售维度）→ 李四自己 + 3 名下属的客户（共 180 人）
4. **L3**：共享 ✓、销售 ✓、交付 ✗
5. **L4**：查看 ✓、编辑 ✓（自己的）、编辑 ○（下属的，默认只读）、导出 ✓
6. **结果**：李四看到 180 行，自己的 48 行可编辑，下属的 132 行默认只读

### 10.3 场景：交付小王在企微侧边栏查看客户

1. 小王角色：`delivery`
2. **L1**：交付侧边栏 ✓、销售侧边栏 ✗
3. **L2**：个人（交付维度）→ 只能看到自己负责交付的客户
4. **L3**：共享 ✓、交付 ✓、销售 ✗ → 侧边栏显示交付视角的用户画像
5. **L4**：查看 ✓、新建跟进 ✓、打标签 ✓
6. **特殊**：如果打开的企微好友不是小王负责的客户 → 侧边栏显示"暂无权限查看该客户信息"

### 10.4 场景：运营总监看全局数据

1. 总监角色：`admin`
2. **L1**：全部功能 ✓
3. **L2**：全公司
4. **L3**：全部字段 ✓
5. **L4**：全部操作 ✓
6. **结果**：看到所有客户、所有字段、可执行所有操作

### 10.5 场景：数据分析师导出报表

1. 分析师角色：`analyst`
2. **L1**：客户列表-查看 ✓、导出 ✓；数据看板 ✓；编辑 ✗、删除 ✗
3. **L2**：全公司（只读）
4. **L3**：共享 ✓、销售 ✓、交付 ✓（分析需要全量字段）
5. **L4**：查看 ✓、导出 ✓，其余 ✗
6. **结果**：看到全量数据，可导出，但所有编辑按钮不可用

## 11. 前端权限控制规范

### 11.1 菜单/路由级

```
用户登录 → 获取角色 → 获取功能权限列表 → 动态生成菜单和路由
无权限的菜单项：不渲染（不是灰色，而是完全不显示）
```

### 11.2 按钮/操作级

```
有权限：正常显示
无权限：不渲染（推荐）或 灰色禁用 + Tooltip "暂无权限"
```

**规则**：
- **危险操作（删除、批量操作）**：无权限时**不渲染**
- **常规操作（编辑、导出）**：无权限时**灰色禁用** + Tooltip
- **原因**：让用户知道这个功能存在但自己没权限，减少困惑

### 11.3 数据范围

```
列表查询时带上当前用户 ID 和数据范围参数
后端根据权限过滤数据行，前端不做客户端过滤（防绕过）
```

### 11.4 字段级

```
根据字段权限配置，动态控制表格列的显隐和详情页字段的显隐
敏感字段默认脱敏，点击"查看完整"时校验权限
```

## 12. 异常与兜底

| 场景 | 处理方式 |
|------|---------|
| 用户无任何角色 | 登录后提示"请联系管理员分配角色"，不显示任何菜单 |
| 用户角色被移除 | 下次刷新页面时生效，当前会话内保持旧权限 |
| 角色被删除 | 已分配该角色的用户自动失去对应权限，下次登录生效 |
| 访问无权限页面（URL 直接访问） | 重定向到 403 页面，提示"暂无权限访问此页面" |
| 操作无权限数据（API 层） | 返回 403，前端显示 Toast "操作失败：暂无权限" |
| 超级管理员误删自己的权限 | `super_admin` 角色不可移除、不可修改 |

role_field_permissions（角色-字段权限）

├── field_group    # shared / sales / delivery
├── visible        # true / false

user_roles（用户-角色关联）
├── user_id
├── role_id
└── assigned_at
