## 1. 背景与目标

### 1.1 背景

当前客户管理页面仅展示**销售侧**数据（专属销售、销售好友状态、客户来源等）。交付组也通过企微与客户建立好友关系，但交付侧数据（专属服务人员、会员状态、社群归属等）无法在同一页面查看，导致：

- 销售无法快速了解客户的交付进度和会员状态
- 管理层无法在一个视图中掌握客户的全生命周期数据
- 交付续费相关的客户筛选需要跨系统操作

### 1.2 目标

在现有客户管理列表页和客户详情弹窗中，**增加交付侧数据展示**，以企微好友关系为数据维度，实现销售 + 交付数据的统一视图。

### 1.3 核心设计原则

- **数据维度**：以企微好友关系（1V1）为列表主维度
- **一客户多行**：同一客户可能有多条企微好友关系（销售 + 官网交付 + 航海家交付），每条关系展开为独立行
- **列表精简**：交付侧仅在列表展示关键字段，完整数据在详情抽屉中查看

## 2. 数据模型

### 2.1 企微好友关系类型

| 关系类型 | 说明 | 对应人员字段 |
|---------|------|------------|
| **销售** | 客户与销售企微号的好友关系 | 专属销售 |
| **官网交付** | 客户与官网交付企微号的好友关系 | 专属服务 |
| **航海家交付** | 客户与航海家交付企微号的好友关系 | 专属服务 |

> 一个客户可同时拥有 1~3 条企微好友关系。官网和航海家由不同交付人员负责。

### 2.2 好友状态（通用字段）

好友状态不区分销售/交付，统一描述该条企微好友关系的当前状态。

| 状态值 | 说明 |
|-------|------|
| 好友 | 双向好友关系正常 |
| 双向好友 | 同上（销售侧用语） |
| 客户删除 | 客户主动删除了企微好友 |
| 随身删除 | 员工侧删除了企微好友 |
| 未添加 | 尚未建立好友关系 |

### 3.3 行展开规则

同一客户按企微好友关系展开为多行：

| 场景 | 行数 | 示例 |
|------|------|------|
| 仅有销售好友 | 1 行 | 销售行，交付列为 — |
| 销售 + 官网交付 | 2 行 | 第 1 行销售，第 2 行官网交付 |
| 销售 + 官网交付 + 航海家交付 | 3 行 | 第 1 行销售，第 2~3 行交付 |

**视觉规则**：
- 同一客户的第 1 行（销售行）顶部有加粗分隔线
- 第 2、3 行（交付行）的客户信息列淡化显示（opacity 0.4），避免视觉重复
- 第 2、3 行的星球ID 显示 —，共用字段（客户等级、客户来源等）淡化

### 3.4 关系类型筛选逻辑

- 选择「销售」→ 仅显示销售行（交付行隐藏）
- 选择「官网交付」→ 仅显示官网交付行
- 选择「航海家交付」→ 仅显示航海家交付行
- 不选 → 展示所有行（默认）

### 3.5 分页

分页计数以**行数**为单位（非客户数）。如一个客户展开为 3 行，占 3 条记录位。

## 4. 客户详情（右侧抽屉）

### 4.1 交互方式

| 属性 | 值 |
|------|-----|
| 触发 | 点击列表「详情」按钮 |
| 形式 | 右侧抽屉（Drawer） |
| 宽度 | 900px |
| 动画 | 打开：从右侧滑入 250ms ease-out；关闭：向右滑出 200ms ease-in |
| 遮罩 | 黑色 80%（`--color-overlay`），点击遮罩关闭 |
| 关闭 | 右上角 × 按钮 / 点击遮罩 |

### 4.2 布局结构

抽屉内部分为左右两栏：

| 区域 | 宽度 | 内容 | 滚动 |
|------|------|------|------|
| 左侧边栏 | 300px | 客户头像/标签/群聊 | 独立滚动 |
| 右侧主区 | 自适应 | Tab 页内容 | 独立滚动 |

### 4.3 左侧边栏

#### 4.3.1 客户概要
