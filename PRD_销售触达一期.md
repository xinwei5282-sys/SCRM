## 1. 文档概述

### 1.1 产品背景

当前系统已经具备客户标签与模板库能力，但尚未形成可执行的销售触达链路。销售和运营在实际工作中仍需要依赖人工筛人、手动组织内容、线下跟踪执行结果，存在以下问题：

- 无统一触达入口，任务创建与执行分散
- 无固定场景的人群自动命中能力
- 无基础防骚扰机制，容易重复打扰客户
- 无结果回收与复盘页面，无法评估触达效果

考虑到一期目标是尽快让触达能力可用，本期不建设通用人群配置器，也不依赖模板变量渲染，而是先以固定场景、手动填写内容的方式，将销售触达主流程打通。

### 1.2 产品目标

- 提供固定场景的触达任务创建能力
- 系统按预设规则自动命中目标客户并完成防骚扰排除
- 支持销售/运营手动填写触达内容并发起任务
- 支持查看任务状态、执行结果与客户回复情况
- 为后续人群配置、模板联动、自动旅程预留数据结构

### 1.3 核心指标

| 指标 | 目标值 |
|------|--------|
| 单个触达任务创建时长 | < 3 分钟 |
| 任务命中人数预览响应时间 | < 5 秒 |
| 运营可独立完成触达任务创建 | 100% |
| 任务结果可追溯率 | 100% |

### 1.4 范围

**In-Scope（本期）**：

- 固定场景触达任务创建
- 系统自动命中人群
- 手动填写触达内容
- 立即发送 / 定时发送
- 任务列表与状态管理
- 结果详情与基础效果统计
- 基础防骚扰拦截与拦截记录

**Out-of-Scope（后期）**：

- 人群规则配置器
- 模板变量渲染与模板联动
- 自动化旅程 / SOP 编排
- A/B 测试
- 审批流
- 智能推荐发送时间
- 复杂转化归因

## 3. 整体方案

### 3.1 方案原则

一期采用“固定场景 + 手填内容 + 任务执行 + 结果回收”的方案，不做过度平台化设计。

整体链路如下：

`选择场景 -> 自动命中客户 -> 防骚扰排除 -> 手动填写内容 -> 创建任务 -> 执行发送 -> 回收结果 -> 查看效果`

### 3.2 模块结构

本期只建设 4 个核心模块：

1. `触达任务创建`
2. `触达任务列表`
3. `触达结果详情`
4. `防骚扰与拦截记录`

### 3.3 设计约束

- 人群规则不支持前台配置，由系统按场景固定计算
- 推送内容不依赖模板渲染，由用户手动填写
- 任务执行时必须保存目标客户快照，避免结果复盘失真
- 防骚扰规则一期固定，不开放策略后台

## 5. 场景规则

本期固定场景由服务端计算，不对外开放配置。

### 5.1 新线索首触达

规则建议：

- 最近 `X` 天新增客户
- 尚未发生过首轮触达
- 客户状态正常
- 不在防骚扰冷却期

### 5.2 沉默客户唤醒

规则建议：

- 最近 `X` 天未回复
- 近 `Y` 天有过互动记录
- 当前非已成交 / 已关闭
- 不在防骚扰冷却期

### 5.3 商机推进跟进

规则建议：

- 商机状态为跟进中 / 待推进
- 最近 `X` 天无跟进动作
- 存在负责人
- 不在高频触达限制内

### 5.4 活动通知

规则建议：

- 命中固定活动标签集合
- 未被本活动触达过
- 客户状态正常
- 不在防骚扰冷却期

> 注：`X / Y` 等阈值一期写入服务端配置，不开放前端修改。

## 6. 防骚扰机制

### 6.1 设计目标

在任务创建和执行前对客户进行基础防骚扰校验，降低重复触达和负向体验风险。

### 6.2 一期固定规则

硬拦截规则：

- 最近已触达
- 超过日频限制
- 超过周频限制
- 连续未回复进入冷静期
- 已拉黑 / 已删除 / 已拒收
- 本活动已触达
- 夜间禁发

### 6.3 规则输出

系统需支持：

- 返回客户是否可触达
- 返回单个客户命中的一个或多个拦截原因
- 任务创建页展示排除人数与原因分布
- 任务详情中保存被拦截客户的快照记录

- 执行中
- 已完成
- 部分失败
- 执行失败

## 10. 技术考虑

### 10.1 执行链路

- 触达任务创建时先计算命中客户，再执行防骚扰排除
- 任务提交后生成客户快照与执行记录
- 发送结果和回复结果需支持异步回流

### 10.2 平台约束

- 一期需兼容当前企业微信 / 现有发送链路能力
- 若发送通道存在官方限制，则系统侧以任务管理、执行记录、结果回收为主，不在 PRD 中承诺超出当前通道能力的自动发送能力

### 10.3 可扩展性预留

- 场景能力后续可升级为可配置策略
- 手动填写内容后续可接入模板库与变量渲染
- 当前结果表结构需支持后续增加点击、转化等指标

## 11. 风险与依赖

### 11.1 依赖项

- 客户基础信息与负责人数据可用
- 标签数据可供活动通知场景使用
- 发送通道可以返回基础执行结果
- 回复结果可被系统回收

### 11.2 风险点

| 风险 | 说明 | 应对方式 |
|------|------|----------|
| 场景规则过粗 | 固定规则可能与业务实际不完全匹配 | 一期先保证能用，二期开放配置 |
| 发送链路能力不一致 | 不同通道返回结果粒度不同 | 结果字段按最低公共能力设计 |
| 防骚扰规则过严或过松 | 可能影响命中规模或客户体验 | 一期保守拦截，保留日志便于调优 |
| 手填内容质量不稳定 | 不同成员内容质量差异大 | 一期通过场景说明和培训约束 |

#### 14.1.3 页面交互

1. 初始进入页面时，页面展示顶部工作台区和空表单；流程条默认高亮第 1 步“任务定义”
2. 用户选择触达场景后，系统展示该场景的人群规则说明，并同步更新任务快照中的“场景”
3. 用户切换发送方式后，系统同步更新任务快照中的“执行方式”
4. 用户点击 `预览人数` 后，系统计算并返回：
   - 命中人数
   - 排除人数
   - 排除原因分布
5. 预览完成后，任务快照同步更新“预计命中 / 防骚扰排除”，并允许保存草稿或提交执行
6. 若用户切换场景、执行人范围，需清空上一次预览结果并提示重新预览
7. 选择定时发送时，展示发送时间字段
8. 选择指定销售时，展示销售成员选择器
9. 各分区标题需展示步骤编号和状态标签，帮助用户快速判断当前完成度

#### 14.1.4 校验规则

| 字段 | 校验规则 | 提示文案 |
|------|----------|----------|
| 任务名称 | 非空 | 请输入任务名称 |
| 触达场景 | 必选 | 请选择触达场景 |
| 推送内容 | 非空 | 请输入推送内容 |
| 推送内容 | 长度 <= 2000 | 推送内容不能超过 2000 字 |
| 发送时间 | 定时发送时必填 | 请选择发送时间 |
| 发送时间 | 不得早于当前时间 | 发送时间不能早于当前时间 |
| 执行人范围 | 指定销售时至少选 1 人 | 请选择执行成员 |

#### 14.1.5 提交逻辑

- 点击 `保存草稿`
  - 表单通过基础校验后保存
  - 任务状态为 `草稿`
- 点击 `提交执行`
  - 要求表单通过全部校验
  - 如未预览人数，先提示用户预览
  - 如为立即发送，任务进入 `待执行`
  - 如为定时发送，任务进入 `待执行`

#### 14.1.6 页面提示

| 场景 | 提示文案 |
|------|----------|
| 未预览人数直接提交 | 请先预览命中人数后再提交 |
| 排除人数较高 | 当前排除人数较高，请确认场景规则和发送时机 |
| 夜间定时发送 | 当前发送时间处于禁发时段，无法提交 |
| 页面停留在长表单中部 | 顶部工作台区持续展示任务快照，便于用户回看当前任务状态 |

#### 14.1.7 异常场景

- 场景规则计算失败：提示“人群预览失败，请稍后重试”
- 无命中客户：允许保存草稿，不允许提交执行
- 全量被拦截：允许查看拦截原因，不允许提交执行

## 16. 执行时序图与状态机细化

### 16.1 核心参与方

一期触达链路涉及以下核心角色与系统：

- `用户端`：销售 / 销售运营
- `触达任务服务`：负责任务创建、状态流转、日志记录
- `场景规则服务`：负责按固定场景计算命中客户
- `防骚扰规则服务`：负责客户排除与拦截原因计算
- `发送执行服务`：负责将任务下发到实际发送链路
- `结果回流服务`：负责接收发送结果与回复结果

### 16.2 预览人数时序

适用于用户在创建页点击 `预览人数`。

```text
用户端
  -> 触达任务服务：提交 sceneCode、executorScope
触达任务服务
  -> 场景规则服务：按场景计算命中客户
场景规则服务
  -> 触达任务服务：返回原始命中客户列表
触达任务服务
  -> 防骚扰规则服务：逐个客户执行拦截校验
防骚扰规则服务
  -> 触达任务服务：返回可触达客户与拦截原因
触达任务服务
  -> 用户端：返回 hitCount、blockedCount、blockedReasonStats
```

#### 16.2.1 结果口径

- `hitCount`：场景规则命中的原始客户数
- `blockedCount`：被防骚扰规则拦截的客户数
- `availableCount`：最终可触达客户数，前端可选展示

#### 16.2.2 异常处理

- 场景规则服务异常：返回“预览失败”
- 防骚扰规则服务异常：返回“排除规则计算失败”
- 命中人数为 0：前端展示空结果，不允许提交执行

### 16.3 创建任务时序

适用于保存草稿和提交执行。

```text
用户端
  -> 触达任务服务：提交任务表单
触达任务服务
  -> 校验服务：基础字段校验
触达任务服务
  -> 场景规则服务：重新计算命中客户
触达任务服务
  -> 防骚扰规则服务：重新执行拦截校验
触达任务服务
  -> 数据库：写入 touch_task
触达任务服务
  -> 数据库：写入 touch_task_target 快照
触达任务服务
  -> 数据库：写入 touch_task_log
触达任务服务
  -> 用户端：返回 taskId、taskStatus
```

#### 16.3.1 设计要求

- 提交时必须重新计算一次命中结果，不能直接复用前端预览结果
- 任务提交时即固化客户快照
- 草稿任务也建议允许保存基础表单信息，但不强制保存完整客户快照
- 正式提交执行时必须保存完整客户快照

#### 16.3.2 草稿与提交差异

| 动作 | 是否重算人群 | 是否保存客户快照 | 任务状态 |
|------|--------------|------------------|----------|
| 保存草稿 | 否/可选 | 否 | 草稿 |
| 提交执行 | 是 | 是 | 待执行 |

### 16.4 立即发送执行时序

适用于任务类型为 `立即发送`。

```text
触达任务服务
  -> 发送执行服务：下发任务
发送执行服务
  -> 数据库：写入 touch_task_execution（待执行）
发送执行服务
  -> 实际发送链路：触发发送
实际发送链路
  -> 发送执行服务：返回发送受理结果
发送执行服务
  -> 数据库：更新 touch_task_execution
发送执行服务
  -> 触达任务服务：返回任务执行进度
触达任务服务
  -> 数据库：更新 touch_task 状态为执行中
```

#### 16.4.1 状态变化

- 创建成功后：`待执行`
- 执行服务开始处理：`执行中`
- 所有目标处理完成后：进入 `已完成 / 部分失败 / 执行失败`

### 16.5 定时发送执行时序

适用于任务类型为 `定时发送`。

```text
用户端
  -> 触达任务服务：提交定时任务
触达任务服务
  -> 数据库：写入任务状态 = 待执行
调度器
  -> 触达任务服务：扫描到期任务
触达任务服务
  -> 发送执行服务：下发任务
后续链路同立即发送
```

#### 16.5.1 调度要求

- 调度器按分钟级扫描待执行任务
- 超过执行时间未发送的任务需记录异常日志
- 已取消任务不可再进入发送链路

### 16.6 结果回流时序

适用于发送结果、回复结果的异步更新。

```text
发送链路 / 外部回调
  -> 结果回流服务：提交发送结果或回复结果
结果回流服务
  -> 数据库：更新 touch_task_result
结果回流服务
  -> 数据库：更新 touch_task_execution
结果回流服务
  -> 触达任务服务：触发任务聚合计算
触达任务服务
  -> 数据库：更新任务汇总指标与任务状态
触达任务服务
  -> 数据库：写入 touch_task_log
```

#### 16.6.1 结果回流范围

- 发送成功 / 失败
- 回复状态更新
- 回复时间更新

#### 16.6.2 聚合逻辑

- 若全部客户均成功：任务状态更新为 `已完成`
- 若部分成功部分失败：任务状态更新为 `部分失败`
- 若全部失败：任务状态更新为 `执行失败`
- 回复结果不改变任务是否完成，但会更新结果指标

### 16.7 取消任务时序

适用于尚未实际进入发送链路的任务。

```text
用户端
  -> 触达任务服务：取消任务
触达任务服务
  -> 状态校验：仅允许取消草稿 / 待执行任务
触达任务服务
  -> 数据库：更新 touch_task 状态 = 已取消
触达任务服务
  -> 数据库：写入 touch_task_log
触达任务服务
  -> 用户端：返回取消成功
```

#### 16.7.1 取消限制

- `执行中`、`已完成`、`部分失败`、`执行失败` 不允许取消
- 调度器触发前取消的定时任务，不得再被执行

### 16.8 状态机细化

#### 16.8.1 任务状态机

```text
草稿
  -> 待执行      （提交执行）
  -> 已取消      （取消）

待执行
  -> 执行中      （开始执行）
  -> 已取消      （取消）

执行中
  -> 已完成      （全部成功）
  -> 部分失败    （部分成功部分失败）
  -> 执行失败    （全部失败）

已完成 / 部分失败 / 执行失败 / 已取消
  -> 终态
```

#### 16.8.2 客户级执行状态机

建议每个客户目标单独维护执行状态，便于结果页精确展示。

```text
待执行
  -> 已拦截      （命中防骚扰规则）
  -> 发送中      （进入发送链路）

发送中
  -> 成功        （发送成功）
  -> 失败        （发送失败）

成功
  -> 已回复      （收到客户回复）
```

#### 16.8.3 成员级执行状态

成员级执行状态主要用于统计页展示，不必做复杂状态机，建议保留以下口径：

- 未开始
- 执行中
- 已完成
- 部分失败

### 16.9 日志记录要求

以下关键节点必须写入 `touch_task_log`：

- 创建任务
- 保存草稿
- 提交执行
- 人群预览完成
- 防骚扰拦截完成
- 开始执行
- 执行完成
- 执行失败
- 取消任务
- 结果回流更新

### 16.10 对账与补偿建议

为避免发送链路与任务状态不一致，建议增加最小补偿机制：

- 定时扫描 `执行中` 且长时间未更新的任务
- 定时扫描 `发送中` 且未回流结果的客户记录
- 对异常任务写入日志并支持人工重试
